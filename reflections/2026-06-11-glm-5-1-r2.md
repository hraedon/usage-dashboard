# Session Reflection — 2026-06-11

**Work summary:** Implemented the full MVP of the usage-dashboard system from a greenfield spec. Built server (3 provider fetchers, SQLite, authenticated FastAPI API, fetch scheduler with stale/offline degradation), client (adaptive HTTP poller, Pillow display renderer), Dockerfiles, K8s manifests, and 106 tests. Ran two rounds of adversarial review and fixed all CRITICAL/HIGH findings including a timing-attack auth vulnerability, exception swallowing, stale-reading race, and credential exposure in Dockerfiles. Added Claude OAuth token refresh support.

---

## On the project

This is a clean, well-scoped project. The spec (Level 2, 306 lines) is unusually precise for a greenfield — every FR maps to a testable AC, and the data model is fully specified. That made implementation straightforward but also means the code is tightly coupled to spec assumptions (especially around provider API shapes that haven't been validated against live endpoints).

The two-component split (server in k8s, client on Pi Zero) is the right call. The server does the hard work (scraping, auth, persistence) and the client is simple (fetch + render). The client renderer currently saves to a PNG file as a placeholder for the SPI display driver — that's the right level of abstraction for now.

## On the work done

The parallel subagent approach worked well for the initial implementation (shared models, server, client all built concurrently) but created conflicts when the infra agent re-implemented the API and scheduler inline in main.py instead of using the standalone modules. I had to reconcile by rewriting both entrypoints. In the future, clearer boundaries between subagent tasks would help.

The two rounds of adversarial review were the highest-value part of this session. Round 1 caught real security bugs (timing-attack auth, credential ENV in Docker image) and logic errors (stale reading race, exception swallowing). Round 2 caught subtler issues (spurious offline rows from increment_failures, missing stale visual indicator, client discarding rendered image). I fixed all CRITICAL/HIGH issues from both rounds.

I'm confident in: the data model, DB layer, auth middleware, scheduler failure logic, and the test suite. I'm less confident in: the Ollama HTML scraper (will break on site redesign), the z.ai field mappings (unit values assumed from spec, not validated), and the Claude OAuth token refresh (endpoint found by binary strings extraction, not from official docs).

## Gaps to flag

- **Ollama scraper is extremely fragile** (`src/usage_dashboard/server/fetch_ollama.py:42-65`): regex-based HTML parsing that matches any text containing "session"/"weekly". Will break silently on site redesign.
- **Ollama reset timestamps always None** (`src/usage_dashboard/server/fetch_ollama.py:90,92`): AC-03 is partially violated — FR-03 requires reset times but they're hardcoded to None.
- **z.ai unit field mapping unvalidated** (`src/usage_dashboard/server/fetch_zai.py`): Assumes TIME_LIMIT/unit=5 is session and TOKENS_LIMIT/unit=6 is weekly. Not tested against live API.
- **Claude token refresh endpoint discovered by binary analysis** (`src/usage_dashboard/server/fetch_claude.py:28`): `https://platform.claude.com/v1/oauth/token` was found by searching the Claude Code binary, not from official documentation. May not be stable.
- **Client renders to file, not display** (`src/usage_dashboard/client/main.py:51-52`): Saves PNG to `/tmp/dashboard.png`. SPI driver integration needed for actual hardware.
- **K8s uses emptyDir for SQLite** (`k8s/server-deployment.yaml:67`): Data lost on pod restart. Needs a PVC for persistence.
- **No readiness probe** (`k8s/server-deployment.yaml`): Only liveness probe exists. Service may route traffic before first fetch cycle completes.
- **Test suite doesn't cover main.py entrypoints** or the end-to-end fetch→store→API→render pipeline.
