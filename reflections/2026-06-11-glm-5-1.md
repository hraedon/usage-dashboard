# Session Reflection — 2026-06-11

**Work summary:** Ran a full socratic specification session for a multi-provider AI usage dashboard. Produced two spec artifacts (spec.md, spec.yaml) at Level 2 (Verifiable), targeting Level 3. The spec covers a two-component system: a k8s-hosted data collection server and a Pi Zero display client showing session/weekly usage for Claude, z.ai, and Ollama.

---

## On the project

This is a greenfield project — just a scaffold with an empty `src/` and placeholder directories. The spec is the first real artifact. The project structure is clean and conventional (Python, hatchling, pytest, ruff, mypy). The `AGENTS.md` is a good skeleton but has a TODO for describing the project — that should be filled in once implementation starts. The "Hard rules" section already has a good guard: "Spec acceptance criteria are the boundary." That'll matter because the spec intentionally defers some research items.

The core architectural challenge is Ollama. Claude and z.ai have real APIs (even if Claude's is flaky). Ollama has nothing official — we're scraping HTML after programmatic login. That's the fragile point in the system and the one most likely to break silently.

## On the work done

The socratic specification process worked well here. The user had clear intent but the data source details were initially vague. The iterative questioning uncovered:
- Claude's OAuth usage API (adapted from CodexBar's research)
- The z.ai quota endpoint with its unclear `unit` field mappings
- The Ollama scraping challenge and the user's preference for self-contained operation over manual cookie refresh

The spec is solid at Level 2. The three "needs research" items preventing Level 3 are genuine — they require running code against live APIs, which can't be resolved through elicitation. The user accepted this honestly.

One thing I'd want a second pair of eyes on: the dependency hints between spec.md and spec.yaml. An adversarial reviewer caught discrepancies (the YAML had independently-derived dependency relationships that didn't match the markdown). I reconciled them but the dependency graph is based on inference, not verified implementation constraints — the spec says as much.

## Gaps to flag

- **Ollama scraping fragility** (`docs/spec.md` FR-03): HTML scraping of ollama.com will break when they redesign. No API exists. The workaround project (ollama-usage) acknowledges this. Worth filing a breadcrumb once implementation starts if Ollama hasn't shipped a usage API.
- **Claude OAuth API reliability** (`docs/spec.md` FR-01): The `/api/oauth/usage` endpoint has a history of 429s (GitHub issue #31021, closed as not planned). The spec handles this with stale readings, but the implementing agent should validate current reliability before committing to this as the primary path.
- **z.ai `unit` field meanings are unknown** (`docs/spec.md` Open Questions): The API response includes `unit: 3`, `unit: 5`, `unit: 6` with no documented mapping. The spec assumes `TIME_LIMIT unit=5` is the session window, but this needs empirical confirmation.
- **No implementation code exists yet** — `src/usage_dashboard/__init__.py` is empty. The spec is the only artifact.
- **AGENTS.md "What this is" is still a TODO** (`AGENTS.md:5-7`): Should be filled in with a project description now that the spec exists.
