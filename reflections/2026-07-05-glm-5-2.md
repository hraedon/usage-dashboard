## On the project

The codebase is in a much better state than when we started. The biggest structural improvement is WI-002: the readings table was single-row-per-provider with upsert-on-conflict, silently destroying usage history the spec assumed would be retained. That's now append-only with a prune job. The ollama cookie persistence (WI-017) brings ollama to parity with Claude's TokenStore pattern. Retiring the PIL renderer (WI-004) removed 399 lines of dead code that was a readability NFR liability nobody could test against (the hardware pivoted to pygame months ago).

The project is well-structured for its scope: small surface, clear separation, good test coverage. The main risk is the SQLite schema migration path — it's idempotent and handles the old schemas, but it hasn't been battle-tested against a real PVC with months of data. The prune-protects-latest-row clause is the safety net.

## On the work done

The ollama cookie persistence (WI-017) was the most satisfying fix — it mirrors an existing, proven pattern (_resolve_claude_tokens) and the adversarial review caught a real bug (stale-reading preserving the "re-login" detail through make_stale_reading). The append-only schema (WI-002) was the biggest change; the subagent handled it well, and the adversarial review caught the prune-deletes-latest regression and the store_reading footgun (consecutive_failures parameter silently resetting provider_state).

The install.sh re-exec fix (WI-011) was tricky — the first version had a subtle bug (trap set before exec is lost because exec replaces the process). The adversarial review caught it. This is the kind of bash subtlety that's easy to get wrong and hard to test.

What I'm less confident in: the DB migration handles the known old schemas but I haven't verified it against a real production PVC. The `</section>` parser (WI-018) works on the synthetic test HTML but ollama.com's actual markup may use different wrapper tags — the `len(tail)` fallback covers this, but it's worth validating against the live page when convenient.

## Gaps to flag

- **The `567d526` commit ("canonical identifier gate")** was already in the working tree at session start — not from this session's work. It adds a CI gate script with zero tests. The script is a security gate (prevents work-domain identifiers from being committed); untested security gates are a risk.
- **k8s/server-deployment.yaml** doesn't declare `RETENTION_DAYS` or `FAILURE_BACKOFF_CAP` env vars. Both default sensibly (7 and 3600), but operators reading the manifest won't know they exist.
- **The `_block_after` `</section>` fallback to `len(tail)`** means if ollama.com ever wraps the weekly section in a `<div>` instead of `<section>`, the block extends to the end of the HTML. This is safe (it'll still find the reset text) but could theoretically capture unrelated content after the usage sections.
- **The ollama cookie is stored in plaintext** in `tokens.json` on the PVC — now with 0600 perms (fixed this session), but still plaintext. Same as Claude tokens. Acceptable for the k8s threat model but worth noting.
- **The `test_initialize_migrates_legacy_schema_without_detail` and `test_initialize_migrates_legacy_schema_without_models` tests** in test_db.py create old schemas that include `consecutive_failures` — they test the migration path but the old-schema-without-throttle-column combination is not explicitly tested (the adversarial review noted this).
